= AGENT.adoc — Review Gradle

_Produit_ : Plugin Gradle de code review assistée par IA — PR review, suggestions, quality gates
_Licence_ : Apache 2.0 (OSS)
_DAG_ : N4 — Plugin Gradle de gouvernance (consomme le code, pas le pipeline)
_Dépendance_ : langchain4j (LLM code review), GitHub API (PR access)

== Principe

Le plugin `review` assiste la **code review de PRs** via IA. Il lit le diff de la PR,
l'analyse avec un LLM (gpt-oss:120b-cloud) et produit un rapport de review structuré :
anomalies, suggestions d'amélioration, adhérence aux patterns du workspace.

Pipeline :
  PR diff ──→ IA (LLM) ──→ rapport de review ──→ commentaires PR ──→ quality gates

Cas d'usage :
- Pre-review IA avant review humaine (gain de temps)
- Vérification des patterns par borough (conventions AGENT.adoc)
- Détection des secrets/tokens (Règle 6 workspace)
- Score de qualité automatique par PR
- Quality gates : review IA passe → humaine facultative ? Non, toujours humaine

== Loi de l'Économie d'Encre — Non-Répétition des Opérations Coûteuses

*PRINCIPE* : Une opération coûteuse (LLM, inférence cloud, analyse IA de PRs, score de qualité automatique) qui a déjà produit un résultat *valide et persistant* NE DOIT PAS être ré-exécutée sur la même entrée. Le résultat existant est la vérité ; le re-calcul est un gaspillage de tokens/crédits API/temps de build. L'IA n'est pas un générateur idempotent gratuit — c'est un service metered. Re-appeler le LLM pour analyser un diff déjà reviewé et dont le rapport est valide : ce geste consomme du token sans produire de nouvelle connaissance. L'économie d'encre est la condition de viabilité économique du pipeline agentique MVP0.

*RÈGLE* :
. Avant toute opération coûteuse, **vérifier l'existence d'un artefact résultat** (rapport de review, score de qualité, cache disque, hash du diff stocké).
. Si l'artefact existe et couvre l'entrée demandée → **réutiliser sans ré-invoquer**.
. Seul le **delta** (nouveaux commits, diff modifié, PR mise à jour) doit être calculé — jamais l'ensemble.
. Ne **jamais écraser** un résultat validé par un re-calcul identique. L'écrasement est un symptôme de non-idempotence économique.
. L'invalidation est **explicite et ciblée** (nouveau commit sur la PR, diff changé), jamais globale par défaut.

*ANTI-PATTERNS SPÉCIFIQUES REVIEW* :
. *Re-review systématique* : re-appeler le LLM sur une PR dont le diff n'a pas changé depuis la dernière review.
. *Re-score inconditionnel* : recalculer le score de qualité sur tous les fichiers parce qu'un seul a été modifié.
. *Cache-busting silencieux* : un `writeText()` qui écrase le rapport de review avant même d'avoir vérifié si le diff a changé.

*PROTECTION* :
. Lecture avant écriture : charger le rapport de review existant avant de re-analyser ; comparer hash du diff (ou SHA du dernier commit) avec hash stocké en metadata du rapport.
. Delta review : ne re-analyser que les PRs dont le diff a changé depuis la dernière review (hash/SHA comparison).
. `DisableCachingByDefault` ne dispense pas de l'idempotence applicative — Gradle ne sait pas ce qu'est une "review déjà valide", seul le domaine le sait.

*RÉFÉRENCE* : Source canonique — `bakery-gradle/bakery-plugin/AGENT.adoc` section "Non-Répétition des Opérations Coûteuses — Loi de l'Économie d'Encre" (Session 153). Portée transverse : 10 boroughs LLM metered. Potentiel 50-80% d'économie LLM sur le pipeline MVP0 complet.

== Tâches Principales

  ./gradlew reviewPr               → analyser une PR (diff + contexte)
  ./gradlew reviewScore            → calculer le score de qualité auto
  ./gradlew reviewGate             → vérifier les quality gates configurées
  ./gradlew reviewReport           → générer le rapport de review (AsciiDoc)
  ./gradlew check                  → tests

== Stack

- Kotlin — DSL Gradle
- langchain4j — LLM code review (gpt-oss:120b-cloud)
- GitHub API (gh CLI wrapper)
- Jackson — parsing du diff / rapport structuré

== Conventions Vérifiées (Workspace)

- Pas de secrets en dur (Règle 6 AGENTS.adoc)
- Conventional commits (feat:, fix:, chore:)
- Patterns par borough (N2 compileOnly vs implementation)
- Zéro import N3 → N2 (violation DAG)
- Pas de publish sans permission

== Ne remplace PAS opencode

Le plugin assiste la review, mais la validation humaine (approval PR)
reste obligatoire. L'IA suggère, l'humain décide.
